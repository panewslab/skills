# PANews Skills 规划文档

## 设计哲学

### Skill 是什么

Skill 是一份**菜谱**，不是 API 说明书。

菜谱告诉你：想做一道菜，需要什么食材、按什么步骤、做出来应该是什么样子。
API 说明书告诉你：这个接口接受哪些参数、返回什么字段。

好的 Skill 应该：
- 以用户的**真实意图**为出发点（"我想了解今天发生了什么"）
- 给出完整的**行动路径**（先做什么、再做什么、遇到情况怎么处理）
- 规定**输出的样子**（给用户看到的结果应该是什么形态）

坏的 Skill（也就是我们现在的实现）：
- 以 API 端点为出发点（"有一个 /search/articles 接口"）
- 只描述单个操作，不描述组合与判断
- 把格式决策留给 Claude 随机发挥

### 目标用户

这不是技术类 Skill。用户不关心 API，不关心参数，他们只关心自己的问题有没有被解决。

## Skill 一：PANews（读者）

### 用户画像

普通人，对加密货币世界感兴趣，但不一定有技术背景。他们可能是：
- 想每天快速了解行业动态的从业者
- 对某个项目或话题感兴趣、想深入了解的人
- 看到一篇文章想读懂它的人

他们不会说"帮我调用 search API"，他们会说：
- "今天加密货币圈有什么大事？"
- "比特币最近怎么了？"
- "帮我看看这篇文章说了什么"

### 核心菜谱

#### 菜谱一：今日速览

**触发意图**：用户想知道今天发生了什么，没有具体话题。

**步骤**：
1. 获取今日每日必读（当天日期）
2. 获取 24 小时热门排行（前 10 条）
3. 综合成一份简报：
   - 开头一句话概括今日市场氛围
   - 列出 3-5 条最重要的新闻，每条附一句话说明为什么重要
   - 结尾标注信息来源时间

**输出原则**：
- 用通俗语言，避免堆砌专业术语
- 如果有术语必须出现，括号里给一句解释
- 不超过 400 字

---

#### 菜谱二：话题深挖

**触发意图**：用户想了解某个具体话题、项目或事件。

**步骤**：
1. 用用户提到的关键词搜索文章（mode=hit，取前 5 篇）
2. 读取每篇文章的标题和摘要，判断相关性
3. 对最相关的 2-3 篇读取全文
4. 综合输出：
   - 这个话题的核心是什么（一段话背景）
   - 最新进展是什么
   - 不同声音或争议点（如有）
   - 推荐延伸阅读（附链接）

**注意**：
- 如果搜索结果为空或相关性低，告诉用户"目前 PANews 上关于这个话题的报道不多"，不要编造内容
- 不要对价格走势做预测

---

#### 菜谱三：读懂一篇文章

**触发意图**：用户给了一篇 PANews 文章的链接或 ID，想了解内容。

**步骤**：
1. 获取文章全文
2. 输出：
   - 一句话核心观点
   - 主要内容分点列出（3-5 点）
   - 如果文章有数据或图表，用文字解释数字的意义
   - 这篇文章为什么值得关注（或不值得）

---

#### 菜谱四：发现热门

**触发意图**：用户想知道现在大家都在关注什么，没有具体话题。

**步骤**：
1. 获取 7 天搜索热词排行
2. 获取 24 小时文章热榜
3. 输出：
   - 本周最热门的 3 个话题是什么，为什么热
   - 今日最受关注的文章（3 条，附简介）

## Skill 二：PANews Creator（创作者）

### 用户画像

在 PANews 上有专栏的内容创作者。他们可能是：
- 独立分析师，定期发布市场分析
- 媒体从业者，需要快速发稿
- KOL，偶尔发布观点文章

他们的典型请求：
- "帮我把这篇文章发到 PANews"
- "我有一篇草稿，帮我检查一下再发出去"
- "帮我看看我上次发的文章审核通过了吗"

### 前置条件

所有操作都需要有效的用户 session。**Session 验证是第一步，永远不能跳过。**

如果用户没有提供 session，第一件事是引导用户获取 session（从浏览器开发者工具中复制），而不是尝试继续。

### 核心菜谱

#### 菜谱一：发布一篇新文章

**触发意图**：用户有内容，想发布到 PANews。

**步骤**：
1. 验证 session 是否有效，获取用户信息和专栏信息
2. 如果用户有多个专栏，询问发布到哪个专栏
3. 如果用户还没有专栏，说明需要先申请专栏，走菜谱"申请专栏"
4. 确认文章信息：
   - 标题（必填）
   - 摘要（必填，建议 50-100 字）
   - 正文（Markdown 格式，系统会自动转换）
   - 封面图（可选，如有本地图片需先上传）
   - 标签（可选，最多 5 个，可以帮用户搜索合适的标签）
   - 语言（默认中文）
5. 将 Markdown 转换为 HTML（使用 md4x 工具）
6. 创建文章（状态：草稿）
7. 给用户看文章预览信息，确认后提交审核
8. 告知用户文章 ID 和预期审核时间

**关键原则**：
- 正文必须由用户提供，不要替用户生成文章内容（除非用户明确要求）
- 先存草稿，用户确认后再提交审核
- 有任何字段不确定，先问用户

---

#### 菜谱二：管理我的文章

**触发意图**：用户想查看自己的文章状态。

**步骤**：
1. 验证 session
2. 获取该专栏下的文章列表，按状态分组展示：
   - 审核中（PENDING）
   - 已发布（PUBLISHED）
   - 草稿（DRAFT）
   - 被拒稿（REJECTED）
3. 如果有被拒稿，主动提示用户可以修改后重新提交

**输出格式**：表格，包含文章标题、状态、创建时间。

---

#### 菜谱三：修改并重新提交文章

**触发意图**：用户想更新一篇草稿或被拒稿，重新提交。

**步骤**：
1. 验证 session
2. 确认要修改哪篇文章（如果用户没说，列出草稿和被拒稿供选择）
3. 获取原文内容展示给用户
4. 根据用户的修改意见，更新对应字段
5. 转换内容格式（如正文有改动）
6. 更新文章
7. 询问是否直接提交审核

---

#### 菜谱四：申请专栏

**触发意图**：用户还没有专栏，想申请开设。

**步骤**：
1. 验证 session
2. 收集申请信息：
   - 专栏名称
   - 专栏简介（建议 100-200 字，说明定位和更新频率）
   - 专栏封面图（需先上传图片）
   - 相关链接（个人主页、社交媒体等，可选）
3. 提交申请
4. 告知用户申请状态和大致审核时间

## Skill 结构规范

### 文件职责

```
SKILL.md        菜谱本身。用意图描述任务，用步骤描述流程，用原则规定边界。
                不出现参数名、字段名、HTTP 方法。

scripts/        工具箱。每个脚本做一件事，输入清晰，输出可预期。
                脚本是实现细节，不是 skill 的主体。

references/     当 SKILL.md 中的步骤需要更多技术细节时，Claude 来查阅。
                例如：字段约束、错误码含义、格式要求。
                不是给用户看的，是给 Claude 在执行时查的字典。
```

### SKILL.md 写作原则

**写这个**：
- 用户说了什么 → Claude 应该做什么
- 先做 A，再做 B，如果遇到情况 X 则做 C
- 输出给用户时应该是什么样子、什么语气
- 什么情况下应该停下来问用户

**不写这个**：
- 接口地址是什么
- 参数叫什么名字
- 返回值的数据结构

### 判断一个 Skill 写得好不好

好的标准：一个不懂技术的产品经理读完 SKILL.md，能明白这个 skill 能帮用户做什么、怎么做到的。

坏的标准：SKILL.md 里出现了 `/search/articles`、`--take`、`PA-User-Session` 这类内容。
